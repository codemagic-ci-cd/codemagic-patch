# Cloudflare setup

:::info
The default [self-host](/docs/setup/self-host) install works without Cloudflare; clients download directly from your storage domain. Use this page when you want a CDN in front of downloads (`DELIVERY_ADAPTER=cloudflare`) or when you plan to use Cloudflare for DNS and TLS on your Patch domains.
:::

Patch integrates with Cloudflare in two common ways:

| Use | Patch setting | When |
| --- | --- | --- |
| **CDN in front of storage** | `DELIVERY_ADAPTER=cloudflare` | Edge caching and automatic cache purge after each release; the usual reason to read this page |
| **R2 object storage** | `STORAGE_ADAPTER=s3` + S3 endpoint | Replace bundled MinIO with managed storage at scale, see [Infrastructure adapters. S3-compatible storage](/docs/setup/infrastructure#s3-compatible-storage) |

The delivery adapter is configured at **initial install** (or by editing `.env.selfhost` before a rerun). Most adapter choices on [Infrastructure adapters](/docs/setup/infrastructure) assume you already have DNS and, for CDN delivery, Cloudflare in front of the storage domain.

## Domains and DNS

Self-host needs two hostnames in the same zone (or with equivalent DNS control):

| Domain | Example | Role |
| --- | --- | --- |
| API / dashboard | `updates.example.com` | `SERVER_URL`, `CodemagicPatchApiUrl` |
| Storage | `storage.updates.example.com` | `PUBLIC_BASE_URL`, `CodemagicPatchDownloadBaseUrl` |

Add **A** or **AAAA** records for both pointing at your Patch host's public IP.

:::info Free Cloudflare plans and nested subdomains
Cloudflare's free Universal SSL certificate covers `example.com` and `*.example.com` only. A storage hostname like `storage.updates.example.com` is a third-level name under the zone, so HTTPS at the edge can fail on a free plan unless you add [Advanced Certificate Manager](https://developers.cloudflare.com/ssl/edge-certificates/advanced-certificate-manager/) (or use a single-level storage hostname such as `updates-cdn.example.com`). Paid plans that include broader edge certificates are the usual choice when you want API and storage grouped under `updates.example.com`.
:::

:::warning Certificate order matters
Caddy obtains Let's Encrypt certificates over HTTP on port 80 during install. If Cloudflare proxies a domain **before** the certificate exists, the HTTP-01 challenge may fail and the install can hang.

**During install:** set the **storage** record to **DNS only** (grey cloud). The API domain can stay DNS-only as well until certificates are issued.

**After install succeeds:** switch the **storage** record to **Proxied** (orange cloud) so traffic flows through Cloudflare. You may proxy the API domain too if you want, that is independent of the delivery adapter.
:::

`PUBLIC_BASE_URL` stays `https://<storage-domain>/codemagic-patch` whether or not Cloudflare is in front. With the delivery adapter enabled, that URL is what clients fetch from and what Patch purges after releases.

## Cache manifests at the edge

Update checks fetch `meta.json` and `manifest.json` on every sync. Bundle artifacts are content-addressed and cached aggressively by default (`cf-cache-status: HIT` after the first fetch).

Manifest JSON is different. Cloudflare does not cache `.json` responses by default, even when the origin sends `Cache-Control: public, max-age=...`. You need both:

1. **Cacheable origin headers** (installer sets this when you pass `--cloudflare`):

```bash
MANIFEST_CACHE_CONTROL=public, max-age=86400
```

2. **A Cloudflare Cache Rule** on your storage domain so manifest and meta JSON are eligible for edge cache. Purge on each release still invalidates them.

### Add a Cache Rule

Cloudflare dashboard → your zone → **Rules** → **Cache Rules** → **Create rule**:

| Field | Value |
| --- | --- |
| Rule name | `Patch manifest JSON` |
| When incoming requests match | Custom filter expression (see below) |
| Then | **Eligible for cache** (Cache Rules → set cache settings → cache: true) |
| Edge TTL | Use cache-control header from origin (respects `max-age=86400`) |

Custom filter expression (adjust the hostname):

```txt
(http.host eq "storage.updates.example.com" and starts_with(http.request.uri.path, "/codemagic-patch/") and http.request.uri.path.extension eq "json")
```

This matches `meta.json`, version-level `manifest.json`, and hash-level `manifest.json` under your bucket prefix.

After you add the rule, publish a release (or republish) so manifests are rewritten with the cacheable `Cache-Control` header, then verify:

```bash
curl -sI "https://storage.updates.example.com/codemagic-patch/<deploymentKey>/meta.json" | grep -iE 'cache-control|cf-cache-status'
curl -sI "https://storage.updates.example.com/codemagic-patch/<deploymentKey>/<binaryVersion>/manifest.json" | grep -iE 'cache-control|cf-cache-status'
```

Run each command twice. The second fetch should show `cf-cache-status: HIT` when caching is working.

If you enable Cloudflare delivery on an existing install, set `MANIFEST_CACHE_CONTROL=public, max-age=86400` in `.env.selfhost`, add the Cache Rule, restart the stack, and publish a release.

## Create a cache-purge API token

The server calls Cloudflare's [Cache Purge API](https://developers.cloudflare.com/api/resources/cache/methods/purge/) after each release. You need a scoped **API Token**: not the account **Global API Key**.

1. Cloudflare dashboard → **My Profile** → **API Tokens** → **Create Token**
2. Use a custom token with:
   - **Permissions:** Zone → **Cache Purge** → Purge
   - **Zone Resources:** Include → the zone that contains your storage domain
3. Create the token and copy it once, this becomes `CLOUDFLARE_API_TOKEN`

The installer verifies the token (active status and purge permission on the zone). Pass `--skip-cloudflare-check` only if you need to bypass verification temporarily.

## Find your zone ID

In the Cloudflare dashboard, open the zone that contains `storage.updates.example.com` (or your storage hostname). The **Zone ID** is on the zone overview page (right column). This becomes `CLOUDFLARE_ZONE_ID`.

## Install with Cloudflare delivery

Run [Self-host install](/docs/setup/self-host#install) with the Cloudflare flags on **first install** (they are written into `.env.selfhost` only when the env file is created):

```bash
scripts/selfhost/install.sh \
  --api-domain updates.example.com \
  --storage-domain storage.updates.example.com \
  --email admin@example.com \
  --github-oauth-client-id Iv1.xxxxxxxxxxxxxxxx \
  --github-oauth-client-secret <github_client_secret> \
  --cloudflare \
  --cloudflare-api-token <cf_cache_purge_token> \
  --cloudflare-zone-id <cf_zone_id>
```

When the installer finishes and `curl` checks pass:

1. Confirm `https://storage.updates.example.com/minio/health/ready` works over HTTPS (storage still DNS-only).
2. In Cloudflare DNS, set the **storage** record to **Proxied**.
3. Add the [Cache Rule for manifest JSON](#add-a-cache-rule) on your storage domain.
4. Re-test downloads; clients should reach MinIO through Cloudflare.

The installer sets:

| Variable | Value |
| --- | --- |
| `DELIVERY_ADAPTER` | `cloudflare` |
| `MANIFEST_CACHE_CONTROL` | `public, max-age=86400` |
| `CLOUDFLARE_API_TOKEN` | Your purge token |
| `CLOUDFLARE_ZONE_ID` | Zone containing the storage domain |
| `CLOUDFLARE_API_BASE_URL` | Optional override (defaults to `https://api.cloudflare.com/client/v4`) |

On each release, Patch resolves public artifact URLs under `PUBLIC_BASE_URL` and purges those paths at the edge. Purge is best-effort, a failed purge does not roll back a release, but devices may see stale bundles until cache expiry. See [How delivery works](/docs/using-patch/delivery).

## Enable Cloudflare delivery after install

If you already have a running stack, edit `.env.selfhost` by hand:

```bash
DELIVERY_ADAPTER=cloudflare
MANIFEST_CACHE_CONTROL=public, max-age=86400
CLOUDFLARE_API_TOKEN=<cf_cache_purge_token>
CLOUDFLARE_ZONE_ID=<cf_zone_id>
```

Add the [Cache Rule for manifest JSON](#add-a-cache-rule), then rerun the installer (it reuses the existing env file) or restart the Compose stack. The installer re-verifies Cloudflare credentials on every run when `DELIVERY_ADAPTER=cloudflare`.

Passing `--cloudflare` on a **rerun** does not update an existing env file; edit the file instead.

## Cloudflare R2 (optional storage)

R2 is S3-compatible. To use it instead of bundled MinIO, set `STORAGE_ADAPTER=s3` and point the S3 variables at your R2 bucket and endpoint, full variable list in [Configuration reference](/docs/reference/configuration). R2 does not replace the delivery adapter: you can still front public downloads with Cloudflare CDN (`DELIVERY_ADAPTER=cloudflare`) while objects live in R2.

That path is manual (not generated by `install.sh`). See [Infrastructure adapters](/docs/setup/infrastructure) for storage and delivery adapter details.

## Verify

```bash
curl -fsS https://updates.example.com/health
curl -fsS https://storage.updates.example.com/minio/health/ready
```

After publishing a release, confirm new bundles are reachable at the CDN URL and that a repeat check picks up the latest manifest (purge working). If devices see an old bundle, check server logs for purge failures.
